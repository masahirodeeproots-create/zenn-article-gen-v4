# Style Guide

## IMPORTANT Rules
<!-- 最大15個。超過時は退役処理 -->

- [voice] です・ます体を基本とし、「だ/である」「体言止め」は禁止。カジュアル表現（「〜なんですよね」「まあ正直」）は自然に混ぜる
  - 追加日: 2026-04-08
  - 根拠: 「だ/である」混在はAI臭の最大要因

- [structure] 冒頭2段構成：「何の話か」→「フック」。3文以内。フックが普遍的なら先に出す
  - 追加日: 2026-04-08
  - 根拠: 読者は最初の3行で読み続けるか判断する

- [density] 記事の最も強い数字（時間削減・コスト・達成率）は冒頭フックか冒頭セクション内に出す
  - 追加日: 2026-04-10
  - 根拠: 中盤初出では離脱読者に届かない

- [structure] 記事全体を貫く「1つの概念的フレーム」を冒頭で明示。テーゼは1つ
  - 追加日: 2026-04-08
  - 根拠: 複数テーゼは散漫になる

- [rhythm] 3文以上同じ文長（短/中/長）が続かないようにする。短→長→短の波を意識
  - 追加日: 2026-04-10
  - 根拠: 同じリズムが続くと読者が離脱する

- [distance] セクション冒頭は出来事・対話・エラーメッセージ・具体的操作で開始。「ここが核心です」等のナレーション宣言禁止
  - 追加日: 2026-04-12
  - 根拠: 宣言型は体験記のベンチマーク記事に存在しない

- [structure] 全セクション等間隔「失敗→原因→解決策」テンプレート禁止。少なくとも1セクションは結論から逆算、1セクションは問題を未解決のまま次へ渡す
  - 追加日: 2026-04-12
  - 根拠: テンプレートが見透けると緊張感が消える

- [density] コードブロック後に必ず1-2文の解説を入れる。コード比率20%超は「貼っただけ」感
  - 追加日: 2026-04-08
  - 根拠: コードだけで伝わるのは著者の幻想

- [distance] 読者への問いかけは1セクション1回まで。「なんか面白くないですか？」連発禁止
  - 追加日: 2026-04-10
  - 根拠: 問いかけ過多はCM感が出る

- [emotion] 失敗談のあとにユーモアを1つ入れると緩和される
  - 追加日: 2026-04-10
  - 根拠: 深刻さだけが続くと読者が疲れる

- [voice] 専門用語は初出時に1文で説明する。括弧注釈は1記事2箇所以内
  - 追加日: 2026-04-08
  - 根拠: 3箇所以上の括弧注釈は教科書臭

## Learned Rules

- [rhythm] セクション冒頭は短文で入る（15字以内）
- [voice] 一人称で書くとき「私は」を連発しない。省略で十分
- [structure] 技術的な説明は「結論→理由→具体例」の順序
- [distance] 他ドメインとの類推（「〜と同じ」）は1記事2-3個まで。7つ並べると「借り物の知識」感
- [rhythm] 段落接続を多様化。「で、」「んで、」だけで全段落をつなげない
- [density] セクション行数比は最長:最短で3:1以上を維持
- [structure] 連番見出し（Round N）を使うなら全要素に見出しを。崩れるなら連番構造を立ち上げない
- [voice] フロントマターは title, emoji, type, topics, published を含める

## Failure Patterns

- [rhythm] 同じ文長の文が4文以上続くと読者が離脱する
- [density] コード比率20%超で「コード貼っただけ」感が出る
- [structure] スペック羅列で冒頭を開始すると読者は「それ何？」と置いてけぼり
- [distance] 普遍的フックより先にシステム説明を置くと没入が1-2文遅延する
- [rhythm] セクション冒頭のナレーション宣言は体験記ベンチマークに存在しない
- [structure] 全セクション等間隔テンプレートは後半でリズムが読めてしまう
