# Zenn記事自動生成システム v4.0 — 改善計画

**作成日**: 2026-04-14
**対象run**: 20260414_100127 / 20260414_140624

## 改善一覧（優先順）

### 1. Writerが毎回ゼロから書き直している

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `run_iteration()` のWriter呼び出しプロンプト |
| **わかりやすく** | Writerは前回の記事を知らずに毎回最初から全文を書いている。だから文体の癖（だ/である混在）が毎回再発する |
| **ワークフローのどこ** | 記事PDCA（article_review）→ Writer呼び出し部分 |
| **なぜ直すべきか** | コンテンツ品質は8.4→8.9→9.1と改善しているのに、文体が4iter全て同じ（です・ます63%）。「修正」ではなく「新規生成」なので指摘が解消されない。HARD FAILが永遠に解除されないデッドロックの根本原因 |
| **どう直すか** | Writerプロンプトに前回の`article.md`を渡し、「以下の記事を修正してください。変更箇所のみ書き換え、それ以外は維持してください」というプロンプトに変更する |

### 2. `--add-dir`によるコスト暴走

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `call_agent()` / `call_agent_async()` の `--add-dir` オプション |
| **わかりやすく** | 毎回のエージェント呼び出しでプロジェクト全体（2.6MB = ペルソナ記事18本+仕様書+過去の実行履歴+全ソース）が丸ごとコンテキストに入っている。style_guide_updaterは20KBのファイル1つだけ必要なのに、毎回1,647Kトークン消費している |
| **ワークフローのどこ** | 全エージェント呼び出しの共通基盤（orchestrator.py の subprocess呼び出し） |
| **なぜ直すべきか** | 1記事$14.34。26回の呼び出しでキャッシュ含め総計1,150万トークン。90%が不要なファイル |
| **どう直すか** | `--add-dir`を削除し、`build_agent_prompt()`で必要なファイルだけプロンプトに埋め込む。ただし13箇所でエージェントが直接ファイルに書き込んでいるため、うち9箇所はorchestrator側でstdoutをファイルに保存する処理の追加が必要（改修規模: 中〜大） |

### 3. Writerプロンプトに「です・ます体必須」がハードコードされていない

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `build_agent_prompt()` のWriter向け部分 |
| **わかりやすく** | style_guide.mdに「です・ます体必須」と書いてあるが、Writerがそれを読んで守るかはLLM任せ。結果、4iter全てでです・ます比率63%（閾値80%）でHARD FAIL |
| **ワークフローのどこ** | 記事PDCA → Writer呼び出し → プロンプト組み立て |
| **なぜ直すべきか** | Reviewerは「コンテンツ品質8.4-9.1」と評価しているのに、文体だけで4.5-5.0に強制される。style_guideを「読むかもしれない」ではなく「読まざるを得ない」形で注入する必要がある |
| **どう直すか** | `build_agent_prompt()`のWriter向け冒頭に以下を直接埋め込む: `「【絶対ルール】全文をです・ます体で書くこと。「だ」「である」「した。」は1文も使わないこと。」` |

### 4. ReviewerがFB構造化JSONを出力しない

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `agent_templates/article_reviewer.md` / `material_reviewer.md` のテンプレート |
| **わかりやすく** | ReviewerにJSON出力を指示しているが、実際は独自のMarkdownフォーマットで出力する。結果、`record_fb_log()`が全iterでパース失敗し、FB差分ロジック（何が治って何が残っているか）が完全に不発 |
| **ワークフローのどこ** | 素材PDCA / 記事PDCA → Reviewer呼び出し → FB構造化ログ記録 |
| **なぜ直すべきか** | `compute_fb_diff()`（resolved/persisted/new/resolution_rateの計算）と `check_fb_stagnation()`（major指摘3回連続残存の検出）が動かない。PDCAの「何が改善されたか」の追跡が不能 |
| **どう直すか** | Reviewerテンプレートに「以下のJSONを必ず含めること。JSONがない場合はレビューとして無効になります」と警告を追加。具体的な出力例を3パターン記載。さらに`run_iteration()`でJSON未検出時にReviewerを再呼び出しするリトライ処理を追加 |

### 5. `agent_memory`の`score_by_axis`が空

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `write_agent_memory()` 関数 |
| **わかりやすく** | 各実行の記録（agent_memory）にOverallスコアしか保存されていない。「冒頭フック: 9.0」「文体: 4.5」のような軸別スコアが空 |
| **ワークフローのどこ** | 振り返りフェーズ → agent_memory書き込み → 次回のEval Designerが参照 |
| **なぜ直すべきか** | 次回のEval Designerが「前回どの軸が低かったか」を参照できない。重みの調整ができず、同じ配分で同じ問題が繰り返される |
| **どう直すか** | `RunState`に軸別スコアの蓄積フィールドを追加。`run_iteration()`で`extract_axis_scores()`の結果をstateに保存。`write_agent_memory()`でYAMLに書き込む |

### 6. `knowledge/`のセマンティック検索が未導入

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `knowledge_store.py` の `filter_by_topic()` |
| **わかりやすく** | 現在はキーワードの単純部分一致で検索している。「マルチエージェント」で検索しても「複数AI」という表現の記事はヒットしない |
| **ワークフローのどこ** | 層3 蓄積基盤 → Strategist/trend_searcherがknowledge/を参照する部分 |
| **なぜ直すべきか** | 記事数が増えると（10記事超）キーワード検索の精度が低下。関連するトレンドや読者ペインを見逃す |
| **どう直すか** | **短期**: TF-IDFベースの関連度スコアリング（追加ライブラリなし）。**長期**: Embeddingベースのセマンティック検索（OpenAI/Anthropic Embedding API + コサイン類似度） |

## 改善の依存関係と推奨順序

```
[1] Writer修正モード化 ──────── 精度改善の根本。これがないとPDCAが機能しない
     ↓
[3] です・ます体ハードコード ── [1]と同時に実施可能。HARD FAIL解消
     ↓
[4] Reviewer JSON出力強制 ──── FB差分ロジックが復活
     ↓
[5] agent_memory充実 ────────── 蓄積精度向上。次回以降に効果
     ↓
[2] --add-dir廃止 ──────────── コスト削減。改修規模が大きいので最後でもよい
     ↓
[6] セマンティック検索 ─────── 長期改善。10記事超で着手
```

**まず[1]+[3]を同時に修正すれば、次回実行でスコアが大幅改善する可能性が高い。**
コンテンツ品質は既に8.4-9.1あるので、文体さえ直ればPDCA成功条件（2連続9.0超）を達成できる。

## 補足: なぜ「コンテンツ品質8.4-9.1なのにスコア4.5」なのか

```
Reviewer評価:
  コンテンツ品質: 8.4/10 → 8.9/10 → 9.1/10（毎iter改善）
  です・ます比率: 63.8% → 63.5% → 63.8% → 62.7%（横ばい）
  
  HARD FAIL発動: です・ます < 80% → Overall強制5.0以下

結果: 4.5 → 4.5 → 5.0 → 4.5（コンテンツは良いのに文体で殺されている）
```

**記事自体は良い。文体が直ればスコアは8.5-9.1になる。** [1]と[3]の修正だけで構造的デッドロックが解消される。
