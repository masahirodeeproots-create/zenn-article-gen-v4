# 記事生成ベンチマーク設計レポート

## 背景

Zenn記事の自動生成品質を定量評価するためのベンチマークを設計した。
本レポートでは、固定ベンチマークの課題と、それを踏まえた設計方針を報告する。

---

## 1. なぜ「固定ベンチマーク」では記事品質を測れないか

### 1.1 記事評価の本質的な難しさ

コード生成やQ&Aのベンチマーク（HumanEval, MMLU等）には「正解」がある。
記事生成には正解がない。同じテーマでも体験記と比較検証では「良い記事」の定義が変わる。

| 評価対象 | 正解の有無 | 固定ベンチマークとの相性 |
|---|---|---|
| コード生成 | テストが通る = 正解 | 高い（HumanEval等） |
| 翻訳 | リファレンス訳あり | 中程度（BLEU等） |
| **記事生成** | **正解なし。「面白さ」は文脈依存** | **低い** |

### 1.2 固定評価軸が機能しない具体例

仮に「文体: 8/10、構成: 7/10、技術深度: 9/10」のような固定軸を作ったとする。

- **体験記**なら「失敗→学びの転換」「開発者の内的独白の自然さ」が品質を決める
- **比較検証**なら「条件統制の明確さ」「定量データの充実度」が品質を決める
- **チュートリアル**なら「再現可能性」「ステップの粒度」が品質を決める

→ 全記事タイプに同じ軸・同じ重みを適用すると、**体験記の"エモさ"も比較検証の"厳密さ"も正しく測れない**。

### 1.3 重みの固定も危険

仮に「冒頭フック」に weight=2.0 を固定すると:
- 冒頭が強い記事 → 高スコア（実態と一致）
- 冒頭は普通だが本文の論理構成が秀逸な記事 → 低スコア（実態と乖離）

人間が読んで「良い」と思う記事がベンチマークで低評価になる状態は、ベンチマーク自体の信頼性を損なう。

## 2. 本システムのベンチマーク設計

上記の課題を踏まえ、**「固定の正解」ではなく「動的な評価基準 + 実例ベースの比較」**で品質を測る3層構造を採用した。

```
┌─────────────────────────────────────────────────────┐
│  層1: ペルソナ記事（human-bench/）                      │
│  → 「良い記事とは何か」の実例集                          │
├─────────────────────────────────────────────────────┤
│  層2: 動的評価基準（eval_criteria.md）                   │
│  → 記事の方向性に応じて毎回生成される評価軸              │
├─────────────────────────────────────────────────────┤
│  層3: スコアリングロジック（Reviewer）                    │
│  → 実例と比較しながら多軸採点 + 加重平均                 │
└─────────────────────────────────────────────────────┘
```

### 2.1 層1: ペルソナ記事ベンチマーク（human-bench/）

**設計思想**: 抽象ルールではなく「実物の良い記事」を基準にする。

```
human-bench/
├── index.yaml          # メタデータ（タイプ・品質・強み）
├── articles/           # 自社の過去記事（15本）
└── top50-ai/           # Zenn AI/LLMカテゴリ Top50記事（50本）
```

index.yaml で各記事に2種類の品質ラベルを付与:

| フィールド | 用途 |
|---|---|
| `material_quality` (high/medium/low) | 素材構造の評価基準として使えるか |
| `style_quality` (high/medium/low) | 文体の評価基準として使えるか |
| `type` (体験記/比較検証/...) | 方向性に応じたフィルタリング |
| `strengths.material` | 素材面で具体的に何が優れているか |
| `strengths.style` | 文体面で具体的に何が優れているか |

**ポイント**: 素材の参考と文体の参考を分離する。
→ 素材構造は優れているが文体は普通な記事（比較検証に多い）も、素材の基準としては活用できる。

### 2.2 層2: 動的評価基準（eval_criteria.md）

**設計思想**: 記事の方向性ごとに「何を測るべきか」自体が変わるなら、評価軸も毎回生成する。

Eval Designer（MetaAgent）が以下を動的に生成:

```markdown
## 素材評価軸（3-5軸）
### S1. 失敗→学びの転換 (weight: 1.5, score: /10)
何を見るか: 単なる失敗報告ではなく、失敗から得た具体的学びが...
副基準:
  - 失敗の具体性（コード/エラーメッセージの引用があるか）
  - 学びの汎用性（読者が自分の開発に適用できるか）
◎ベンチマーク: 「Agent Loop実装記」→ 3回目のリトライで設計方針を変えた描写が...
✗ダメ: 「うまくいきませんでした。次はうまくいきました」← 何も学びがない

## 記事評価軸（4-7軸）
### A1. 冒頭フック (weight: 1.8, score: /10)
...
```

| 設計制約 | 値 |
|---|---|
| 素材主軸数 | 3-5個 |
| 記事主軸数 | 4-7個 |
| 各主軸の副基準 | 2-4個（スコアなし、自由記述FB） |
| 重み範囲 | 0.5 - 2.0 |
| ベンチマーク引用 | 必須。ペルソナ記事の具体箇所を示す |
| ダメパターン | 必須。「こう書くとこうなる」レベルで具体的に |

**学習メカニズム**: 過去の実行記録（agent_memory）から以下を参照して重みを回ごとに調整。

```
優先度: human_feedback > eval_criteria_accuracy > score_by_axis

- score_by_axis:     どの軸が低かったか → 重みを上げるべきか
- eval_criteria_accuracy: 「冒頭フックの重みが高すぎた」等の振り返り
- human_feedback:    AI評価と人間感覚のギャップ（最も信頼性が高い）
```

### 2.3 層3: スコアリングロジック（Reviewer）

Reviewer（Material Reviewer / Article Reviewer）が以下の手順でスコアリング:

```
1. 「この記事の評価で最も重要なこと」を読む → 全体の評価姿勢を設定
2. 各主軸について:
   a. ベンチマーク記事の具体例と比較
   b. アンチパターンに該当していないか確認
   c. X/10 でスコアリング
   d. 副基準ごとに自由記述フィードバック
3. 加重平均スコアを算出:
   Overall = Σ(weight_i × score_i) / Σ(weight_i)
```

加えて、**自動メトリクス**（機械チェック）で客観的な下限を担保:

| メトリクス | 閾値 | 目的 |
|---|---|---|
| コード比率 | ≤ 20% | コードの貼りすぎ防止 |
| です・ます比率 | ≥ 80% | 文体の統一性 |
| セクション行数比 | ≥ 3:1 | 見出しだけスカスカ防止 |
| 連続同長文帯 | ≤ 4文 | 単調なリズム防止 |

## 3. AutoAgentとの思想的つながり

AutoAgent (arXiv 2502.05957) の核心思想:
- **自然言語駆動**: 評価基準を人間がコードで定義するのではなく、AIが自然言語で生成
- **セルフプレイ自己改善**: 生成→評価→改善のループが自律的に回る
- **ゼロコード**: 新しい記事タイプへの対応にコード変更が不要

本ベンチマーク設計はこの思想に沿っている:
- 評価軸の定義自体をAIが毎回行う（固定コード不要）
- ペルソナ記事の追加だけで評価基準が進化する（ゼロコード拡張）
- 人間フィードバックで AI評価 ↔ 人間感覚 のギャップを学習する

## 5. まとめ

| 観点 | 固定ベンチマーク | 本システムの動的ベンチマーク |
|---|---|---|
| 評価軸 | 事前に固定 | 記事の方向性に応じて毎回生成 |
| 基準の定義 | 抽象ルール | 実在する良い記事との比較 |
| 記事タイプ対応 | タイプ別に別のベンチマークが必要 | 1つの仕組みで全タイプに対応 |
| 拡張 | コード変更が必要 | ペルソナ記事の追加のみ |
| 学習 | なし | agent_memory + human_feedbackで自動改善 |
| 弱点 | タイプ間の公平性は高い | AI評価者の品質に依存（人間FBで校正） |
