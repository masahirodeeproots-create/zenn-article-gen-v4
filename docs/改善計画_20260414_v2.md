# Zenn記事自動生成システム v4.0 — 改善計画 v2

**作成日**: 2026-04-14
**前提**: 改善[1]〜[5]適用済み。3回の実行結果を踏まえた次フェーズの改善。

---

## 現状の問題サマリー

| 問題 | 影響 | 根本原因 |
|---|---|---|
| **記事スコアが4.0-5.0で頭打ち** | PDCAが成功終了（9.0超）に到達しない | です・ます比率が71%止まり（閾値80%）。LLMが長文生成中に文体ルールを忘れる |
| **Writer修正モードが機能しない** | 文体の癖が毎回再発 | LLMは「部分修正」が苦手で全文再生成する傾向。プロンプト指示では制御不能 |
| **ADD_AGENTエージェントが未使用** | エスカレーションの効果ゼロ | run_iteration()に動的エージェント呼び出しパスがない |
| **テスト条件が不適切** | システム汎用性が未検証 | 同じお題で3回回していた。別お題での実行が必要 |
| **コスト$12.73** | 1記事あたり高い | プロンプトサイズ最適化の余地あり |

## 改善一覧（優先順）

### [7] 文体修正専用ステップの追加（最高優先）

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `run_iteration()` の記事PDCA内。Writer→Reviewer間に文体修正ステップを挟む |
| **わかりやすく** | LLMに「です・ます体で書け」と指示しても長文で71%が限界。生成後にorchestrator側のPythonコードで「した。」→「しました。」等を正規表現置換する機械的ステップを追加する |
| **ワークフローのどこ** | 記事PDCA: Writer → **【新】文体修正** → Article Reviewer |
| **なぜ直すべきか** | コンテンツ品質は8.0-8.5あるのに、文体2.0/10でHARD FAILされてOverall 4.0-5.0に強制。文体さえ直ればスコア8.5+に跳ねる。**他の全改善より即効性が高い** |
| **どう直すか** | `run_iteration()`内のWriter出力後、Reviewer呼び出し前に`fix_desu_masu(article_text)`関数を追加。正規表現で「だ。」→「です。」「した。」→「しました。」「である。」→「です。」等を置換。compute_desu_masu_ratio()で80%未満なら置換実行 |

### チェックポイント

| # | チェック | 確認方法 |
|---|---|---|
| 7-A | `fix_desu_masu()`関数が定義されているか | grep確認 |
| 7-B | Writer出力後・Reviewer前に呼ばれているか | run_iteration()内のコードフロー確認 |
| 7-C | 置換後のです・ます比率が80%以上になるか | テスト用テキストで`compute_desu_masu_ratio(fix_desu_masu(text))`を実行 |
| 7-D | テスト全パス | pytest |

### [8] ADD_AGENTエージェントの実行パス追加

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `run_iteration()` の記事PDCAフロー |
| **わかりやすく** | エスカレーションでdraft_surgeonやnarrative_puncherが生成されるが、呼ぶコードがないので一度も使われない。Writer→Reviewer→SGUの固定3体しか呼ばれていない |
| **ワークフローのどこ** | 記事PDCA: Writer → Reviewer → SGU → **【新】動的エージェント実行** |
| **なぜ直すべきか** | ADD_AGENTエスカレーションが「エージェント生成して終わり」になっている。生成コスト（$0.24）が無駄 |
| **どう直すか** | `run_iteration()`のSGU呼び出し後に、registryからphase="dynamic"のエージェントを列挙し、存在すれば順次実行する。出力はiterations/{N}/に保存 |

### チェックポイント

| # | チェック | 確認方法 |
|---|---|---|
| 8-A | registryからdynamicエージェントを列挙するコードがあるか | grep確認 |
| 8-B | 動的エージェントの出力がファイルに保存されるか | コード確認 |
| 8-C | dynamicエージェントが0体のときエラーにならないか | 正常分岐確認 |
| 8-D | テスト全パス | pytest |

### [9] 別お題でのテスト実行

| 項目 | 内容 |
|---|---|
| **直すべき部分** | テスト方法自体の問題 |
| **わかりやすく** | 同じソース+同じinstructionで3回回していた。前回の失敗記録を前提にStrategistが「3回目の挑戦」として戦略を立てるので、汎用性の検証になっていない |
| **ワークフローのどこ** | 運用レベル（コード変更不要） |
| **なぜ直すべきか** | 新しいお題で初めて実行したときにシステムが正常動作するか未確認 |
| **どう直すか** | 全ソースファイル（9本）を使い、instruction を変えて実行する。例: `--source source-material --instruction "OshigotoAIのアセット解決パイプラインの比較検証記事。技術的な深掘り中心に。"` |

### [10] プロンプトサイズ最適化

| 項目 | 内容 |
|---|---|
| **直すべき部分** | `build_agent_prompt()` の各エージェント向けファイル埋め込み |
| **わかりやすく** | code_analyzer/dev_simulatorにソースファイル全文（639行）を毎回渡している。Strategistにもagent_memory最大10件を渡している。これらが不要に大きい |
| **ワークフローのどこ** | 全エージェントのプロンプト組み立て |
| **なぜ直すべきか** | コスト$12.73のうち、入力トークンの過大がキャッシュコストを膨張させている |
| **どう直すか** | (a) Writerに全素材原文ではなくサマリー（3,000字）を渡す。(b) agent_memoryの各エントリを要約して渡す。(c) style_guide_updaterを2iter毎に呼ぶ |

| 10-A | summary.jsonのtotal_costが$8以下か | 実行後確認 |
| 10-B | 記事品質が低下していないか | スコアが前回以上か |

### [11] writer_handoff.md自動生成

| 項目 | 内容 |
|---|---|
| **直すべき部分** | 素材PDCA完了後、記事PDCAの前 |
| **わかりやすく** | 素材フェーズで発見した制約（推定出力の注意、概念フレームの必須キーワード、使ってよい/いけない素材の分類）がWriterに伝わっていない |
| **ワークフローのどこ** | `cmd_run()`のワークフロー実行部分。素材PDCAフェーズ完了後にwriter_handoff.mdを自動生成 |
| **なぜ直すべきか** | learning_logに2回連続で「L1: 素材→記事引継ぎ文書を必ず生成する（HIGH）」と記録されている。Strategistが学んでいるのにシステムが対応していない |
| **どう直すか** | 素材PDCA完了後に、最終iterのmaterial_review + strategy.mdの核心メッセージ + 推定出力リストを統合してwriter_handoff.mdを生成。Writerプロンプトに注入 |

## 改善の依存関係と推奨順序

```
[7] 文体修正専用ステップ ─── 最即効。スコア4.0→8.5+への跳躍
     │
[8] ADD_AGENT実行パス ────── エスカレーション効果の発揮
     │
[9] 別お題テスト ─────────── 汎用性検証。コード変更不要
     │
[10] プロンプトサイズ最適化 ─ コスト$12→$8以下
     │
[11] writer_handoff.md ───── 素材→記事の情報損失防止
```

**[7]だけでスコアが劇的に変わる。** コンテンツ品質8.0-8.5は確認済みで、文体（2.0/10）が唯一のボトルネック。正規表現による機械的な文体修正は確実に効く。
